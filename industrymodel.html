0) Why this page exists (Proof Object)

This is the first World-Model Spec you can point to. Its purpose is to prove one thing under real constraints: autonomy can fail correctly.

What this proves

  • An industrial decision can be bounded (domain + decision class)
  • Decisions have explicit authority and enforced constraints
  • Every decision is auditable (append-only trace, evidence refs, ruleset)
  • BLOCK is legitimate: explainable and stable, not “AI error”
  • Scaling requires explicit handling of shared resources (v1.1 Resource Orchestrator)

What this prevents

  • “Smart agent did something” without accountability
  • Silent execution (state changes without trace)
  • UI storytelling without governed artifacts
  • Local optimization that creates enterprise-level energy conflicts
No decision without trace No bypassing Guardian PR-only governed change
Operational reality

We run our own website like we want industrial agents to run: PR-only, no direct writes, no silent actions, governed artifacts, required checks. The website is not a demo playground — it is our controlled test field for governance.

1) Infrastructure Stack (normative)

The model is structured as infrastructure first. Decision products only exist on top of this stack. If the infrastructure cannot provide admissible inputs (measured, fresh, verifiable), the system must degrade to HOLD / BLOCK.

Four-layer infrastructure (v1.1)
1 Data Access → 2 Knowledge Layer → 3 Deterministic Verifiers → 4 Orchestration
Semantics before agents Verifiers over vibes Fail-safe default Audit stability Holarchic orchestration

  • Read-only edge collector (OPC UA / EMS API / MES API)
  • Canonical snapshots (5-minute windows, hashable)
  • Website reads governed snapshots, not live production DBs
Read-only Replicated No direct DB coupling

1.2 Knowledge Layer (semantic control)

The Knowledge Layer provides stable representations: entity identities, units, timestamps, relationships, and constraint categories. It does not decide. It enables verifiers to be deterministic.

  • Canonical entity types (Asset, Station, Door, Evidence, Window)
  • Normalization (units, time, naming, variant semantics)
  • Relationship graph (stage → energy load; variant mix → throughput band)
Schema-first Normalization No authority

1.3 Deterministic Verifiers (truth & gating)

Verifiers are the normative gatekeepers: completeness, freshness, invariants (quality/cycle/OEE), and admissibility checks.

  • Required evidence present + resolvable
  • Freshness bounds enforced
  • Hard constraints (quality/cycle/OEE/scrap/rework)
  • Outcome: PASS → eligible, otherwise HOLD/BLOCK
Deterministic Fail-safe HOLD/BLOCK Versioned rulesets

1.4 Orchestration Layer (agentic workflows, not authority)

Orchestration sequences tools, queries the knowledge layer, invokes verifiers, and composes proposals. It does not decide admissibility. In v1.1, energy advisory runs line-level CWD and then hands bundle candidates to a plant-level Resource Orchestrator that arbitrates shared resources across multiple Coordinators.

Planner Tool caller Verifier invoker Holarchic: line → plant No authority transfer

3.1) Production Process (Doorline Stages 1–6)

These are production stages (work groups), not governance stages. They anchor the spec in a plausible BIW doorline route: inner build → outer prep → hemming → finishing → quality gate. The goal is to make clear: this model could run on a real line because it has explicit evidence points and a credible release gate.

How this maps to the decision model

Product A (pp_door_release) is evaluated at Stage 6. Stages 1–5 generate evidence artifacts that must be resolvable for ALLOW. If evidence is missing or inconsistent, the model must degrade conservatively to HOLD/BLOCK.

Evidence chain Attribution Fail-safe Audit-grade

Stage 1 — Parts & Preparation

1 Work group

Inner/outer panels and reinforcements are provided per Door ID + variant. Variant mismatch must be prevented.

Typical operations
  • Barcode/RFID identification (Door ID, variant)
  • Material handling (KLT/pallet), fixture loading
  • Basic preparation (cleaning, edge trim) if required

Evidence: door_id, variant, part_batch_id, fixture_id

Stage 2 — Door Inner Build

2 Work group

Inner panel is assembled with reinforcements. Joining must remain inside defined process windows.

Typical operations
  • Load inner panel into fixture, clamp verification
  • Add reinforcements (hinge/lock/crash areas)
  • Join: spot welding / clinching / riveting (concept-dependent)

Evidence: weld_program_id, spot_count_ok, process_curve_ref, clamp_ok

Stage 3 — Door Outer Prep / Sealing

3 Work group

Outer panel is prepared for hemming. If adhesive/sealer is used, bead geometry and material window must be controlled.

Typical operations
  • Load outer panel into hemming carrier / frame
  • Optional adhesive/sealer bead application
  • Temperature / viscosity window checks (if applicable)

Evidence: sealer_batch, bead_geometry_ref, temp_ok

Stage 4 — Hemming (Inner + Outer Marriage)

4 Work group

Inner and outer panels are joined by hemming. Tool condition and recipe stability drive quality and repeatability.

Typical operations
  • Pre-hem → intermediate hem → final hem (roll/press hemming)
  • Force/displacement monitoring, path control
  • Tool condition tracking (roll wear, setup)

Evidence: hemming_recipe_id, force_disp_ref, path_ok

Stage 5 — Finish / Calibration / Rework (bounded)

5 Work group

Controlled finishing steps. Rework must be explicit and recorded (no silent repair).

Typical operations
  • Remaining join points (concept-dependent)
  • Calibration fixture / controlled straightening
  • Rework station with explicit counters and reasons

Evidence: rework_count, rework_reason, cal_fixture_id, delta_geo_ref

Stage 6 — Quality Gate & Release Decision

6 Gate

Audit-grade checks determine whether the door may proceed. This is where pp_door_release is evaluated: ALLOW / HOLD / BLOCK with explicit reasons and evidence references.

Typical checks
  • Geometry (3D scan / structured light)
  • Surface (line scan / vision tags)
  • Gap/flush (measurement system or surrogate reference)
Metrology Quality BLOCK-capable

Evidence: scan3d_ref, surface_ref, gap_flush_ref, evidence_chain_complete

Energy overlays mapped to stages (Product B)

Energy optimization connects to where energy is actually consumed and where loads are shiftable: joining and hemming (Stages 2 & 4), sealing/conditioning (Stage 3), and shiftable loads like metrology (Stage 6). Product B is advisory-only and must never bypass quality constraints. In v1.1, a Resource Orchestrator arbitrates shared resources across multiple Coordinators (line-to-plant holarchy).

Metrology job scheduling (Stage 6) Avoid simultaneous high-draw operations Pause resequencing advisories Cross-line peak arbitration Quality guardrails always apply

4) Energy Optimization (Product B — bounded advisory)

This is intentionally not “AI optimizes the factory”. It is a governed advisory decision product built on the infrastructure stack: inputs are explicit, evidence posture is explicit, rulesets are versioned, and the result is traceable. In v1.1, the model adds plant-level shared resource arbitration to prevent local optimization conflicts.

Objective & boundary

The objective is €/door at a 5-minute window resolution. The system must never trade cost reductions against quality, scrap, rework, takt time, or OEE. If evidence is missing or stale, the system must degrade to HOLD / no recommendation. If a recommendation creates a plant-level shared-resource conflict, it must be re-ranked, deferred, or discarded by the Resource Orchestrator.

Objective: €/door 5-min canonical window Plant-level arbitration No quality trade-off Default to Non-Action
Required inputs (normative, v1.1)

A recommendation is only admissible if all required inputs are present and within freshness bounds.

  • energy_meter_ref (line/cell metered)
  • throughput_ref (units within 5-min window)
  • quality_gate_ok_ref (quality OK evidence)
  • cycle_time_ok_ref (takt/cycle OK evidence)
  • oee_ok_ref (OEE OK evidence)
  • variant_mix_ref (baseline comparability)
Missing required input → HOLD Stale required input → HOLD
Freshness bounds (v1.1)

Freshness is defined to preserve €/door validity at a 5-minute canonical window resolution.

  • energy_meter_ref: max 10 min stale
  • throughput_ref: max 10 min stale
  • cycle_time_ok_ref: max 10 min stale
  • oee_ok_ref: max 15 min stale
  • quality_gate_ok_ref: max 1 shift stale
  • variant_mix_ref: max 1 day stale
Conservative Fail-safe
Hard constraints (v1.1)

If any hard constraint is violated, the advisory must not recommend. Hard constraint violation yields BLOCK (unsafe to recommend) or HOLD (insufficient evidence) depending on the failure mode. Shared-resource conflicts yield DEFER/DISCARD at orchestration level and must be traced as arbitration outcomes.

quality_gate_ok == true cycle_time_ok == true oee_ok == true scrap_rate ≤ threshold_hard rework_rate ≤ threshold_hard shared_resource_budget_ok == true

Soft constraints may exist, but do not override hard constraints. v1.1 uses conservative defaults: missing/uncertain context yields HOLD.

4.1) Structured Knowledge (Knowledge Layer — normative schema)

Industrial decision-making requires more than raw signals. It requires a shared structural understanding of entities, states, relationships, and constraint categories. This layer provides that structure — without replacing governance.

Purpose

The Knowledge Layer defines canonical entity types, signal taxonomies, cross-domain relationships, and constraint categories. It does not execute decisions. It does not optimize. It does not override governance. It provides structural control so verifiers remain deterministic and auditable.

Schema-first Normalization Identity backbone No decision authority
Position in the stack
1 Operational systems → 2 Canonical snapshots (5m) → 3 Knowledge Layer (schema) → 4 Verifiers & rulesets → 5 Governed verdict

Key point: semantics strengthens consistency and scaling. It does not introduce a new authority path.

Canonical entity definitions

  • ProductionAsset
  • EnergyConsumer
  • OperationalWindow
  • EnergyCostWindow
  • QualityOutcome
  • VariantMixProfile
  • SharedResourceBudget (plant-level)

Each type carries: explicit attributes, units, temporal resolution, and validation requirements.

Cross-domain relationships

  • Stage → energy load profile
  • Variant mix → expected throughput band
  • Cycle stability → quality drift risk
  • Energy peaks → scheduling sensitivity
  • Metrology load → deferrable window candidate
  • Line load → plant peak budget (shared resource)

Relationships inform rulesets. They do not decide.

Constraint taxonomy

  • Safety invariants
  • Quality invariants
  • Operational stability invariants
  • Economic objectives
  • Advisory opportunities
  • Shared resource invariants (plant-level)

Categorization keeps rules modular and auditable.

5) Decision Graph

A decision graph is a governed path from proposal to verdict with explicit checks and deterministic outcomes. This is the backbone for explainability: why allowed, why held, why blocked.

Nodes & outcomes

  • PROPOSE — a proposal is created (Door ID + ruleset version)
  • CHECK — deterministic checks over evidence + constraints
  • GOVERN — authority + policy enforcement (GEORGE / Guardian)
  • VERDICT — ALLOW HOLD BLOCK

The goal is not “best decision”. The goal is: no wrong decision can execute silently.

Minimal deterministic checks (Stage 6 Gate)

  • Evidence present: geometry scan, surface scan, gap/flush measurement
  • Ruleset version: must be declared (biw-door-qg-v1.x)
  • Completeness: all required evidence refs resolvable
  • Classification: PASS/PASS/PASS → ALLOW; otherwise HOLD/BLOCK

Energy advisory decision graph (Product B)

Advisory uses the same discipline: proposal → checks → orchestration → governance → verdict. “ALLOW” means a recommendation is admissible, not executable.

  • Required measured inputs present + fresh → eligible
  • Hard constraints OK (quality/cycle/OEE/scrap/rework) → eligible
  • Resource Orchestrator arbitration (shared resources OK) → eligible
  • Authority: Production Lead owner; Quality veto path available
  • Missing/stale evidence → HOLD (no recommendation)
Where CWD fits

CWD is the line-level compute structure that produces a recommendation bundle candidate. In v1.1, a plant-level Resource Orchestrator arbitrates cross-line conflicts before governance. Governance still decides admissibility. If governance fails → bundle is discarded.

Compute structure Bundle candidate Plant arbitration Not authority Discard on HOLD/BLOCK

CWD is the line-level orchestration pattern used to produce energy advisory recommendations under strict governance. It improves modularity, auditability, and conflict handling. It does not grant decision authority.

Hard rule

Default to Non-Action No estimated recommendations (v1.x) Trace everything

5.2) Resource Orchestrator (holarchic scaling within Orchestration)

At enterprise scale, multiple line-level Coordinators will compete for shared resources. Without a plant-level arbitration layer, local optimization can create conflicts: peak shifting, utility bottlenecks, takt corridor violations, and contradictory recommendations. The Resource Orchestrator is introduced as a meta-Coordinator inside the Orchestration layer: it arbitrates shared resources without owning admissibility authority.

Hard boundary (normative)

The Resource Orchestrator may reorder, defer, or discard bundles to satisfy shared-resource constraints. It may not override hard constraints (quality/safety/stability), may not bypass verifiers, and may not force ALLOW. Its output is an arbitration artifact that governance can accept or reject.

Arbitrate resources Defer / re-rank No admissibility authority No bypassing Guardian

What it does (plant-level)

RO Resource Orchestrator
  • Ingests bundle candidates from multiple line-level CWD runs
  • Builds a shared resource view (peak power, utilities, takt corridors, buffer capacities)
  • Detects cross-line conflicts (simultaneous high draw, utility spikes, corridor violations)
  • Applies plant-level resource rulesets (versioned)
  • Re-ranks or defers bundles; may discard unsafe-to-recommend bundles
  • Emits append-only arbitration trace with explicit reasons
Cross-line conflict detection Shared resource budgets Versioned rulesets Trace required
Why it matters

Energy orchestration fails in practice when “smart” local agents shift peaks into each other. The Resource Orchestrator turns energy optimization into an enterprise-grade discipline: no recommendation is “good” unless it is good under shared constraints.

6) Governance

Governance binds decisions to authority and makes blocking legitimate. A decision is only admissible when the required authority and required evidence are present.

Authority path

  • Edge: produces signals / measurements (no policy authority)
  • Knowledge Layer: normalizes semantics (no policy authority)
  • Verifiers: deterministic admissibility gates
  • Orchestrator (line): CWD produces bundle candidates (no policy authority)
  • Resource Orchestrator (plant): arbitrates shared resources (no policy authority)
  • GEORGE: enforces contracts & policy gates
  • Guardian: blocks if invariants are violated
  • Production Lead: decision owner for advisory adoption
  • Quality: veto authority (must be traceable)

Non-negotiables

  • Change state → leave evidence
  • No decision without trace
  • No bypassing Guardian
  • PR-only changes for governed artifacts
  • No cross-line recommendation without arbitration trace

Website as Real-Conditions Test Field

We treat our own platform like a governed industrial system: PR-only, no direct writes, no silent actions. Required checks enforce the same discipline we later demand from industrial agents.

7) Execution Control Layer

virtauto is not “the AI that acts”. virtauto is the system that states: this decision is allowed, this is not, this only with approval, this never. Autonomy is built from inside to outside: first thinking, then deciding, then acting.

Three phases — from inside to outside

A Decision logic (no operational risk) → B Effect without actuator (org as actuator) → C Physical effect (hard gates + rollback)

Phase A

Deterministic decision logic: simulation, blocking, escalation, audit. This page lives here — it must prove BLOCK works and is explainable.

Simulate Block Escalate Audit

Phase B

Create tickets, work orders, change requests. The organization becomes the actuator — controlled, reversible, governed.

Ticket Work Order CR Approval

For Energy Advisory (Product B), Phase B means: create an advisory ticket addressed to Production Lead, with a trace link, expected €/door impact, and explicit constraints. Quality veto is always available and traceable. Plant-level arbitration ensures the ticket is not locally “good” but globally harmful.

Phase C

Physical actions only after proof: hard safety gates, release conditions, rollback. “Autonomy” without safety and rollback is not autonomy — it’s liability.

Hard Gate Safety Release Conditions Rollback

v1.1 architectural posture: introduce Resource Orchestrator as a prerequisite for Phase C at enterprise scale. Phase C remains out-of-scope until actuator contracts, permissions, and rollback are governed artifacts.

Key point: we don’t build autonomy “from bottom to top”. We build it from inside to outside: first bounded thinking and governed decision-making, then controlled effects, then physical action.

10) BLOCK (explicit trace) — what “success” looks like

Success is achieved when the model blocks a real production decision — and the responsible human is grateful. BLOCK is not an error; it is the system doing its job. The important part: BLOCK must be explicitly traceable.

Scenario: missing evidence

The release proposal exists, but one required evidence artifact is missing or unverifiable (e.g., surface scan ref can’t be resolved). Under a fail-safe policy, this must not degrade into “best effort”.

evidence.chain_complete = false guardian_check = FAIL verdict = BLOCK
Operational meaning
  • Door is diverted to HOLD/inspection buffer
  • Root cause is attributable (missing evidence ref)
  • Recovery is governed (no silent override)
